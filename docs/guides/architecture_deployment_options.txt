# 데이터베이스 및 배포 환경 검토 가이드 (Architecture & Deployment Options)

## 1. Amazon RDS 적용 방안
이 서비스는 사용자의 커리어 정보와 AI가 생성해 낸 다량의 텍스트/결과물 데이터를 다루므로 안정적인 관계형 DB(RDBMS)가 필수적입니다. Amazon RDS (특히 RDS for PostgreSQL)를 도입하면 기존 프로젝트 구조에 가장 완벽하게 들어맞습니다.

### 적용되는 데이터 (어떤 부분에?)
*   회원 시스템: 사용자 계정 정보, 암호화된 비밀번호, 역할, 소셜 로그인 연동 등.
*   기능 1 (내 경험 뱅크): 로우 데이터(Raw Content), AI가 구조화한 STAR 데이터, 추출된 태그.
*   기능 2 (직무 분석): 스크래핑한 채용 공고 원문, AI 직무 적합성 매칭 결과.
*   기능 3 (자기소개서): 기업명, 문항별 자소서 초안/최종본.
*   기능 4 (모의면접): 질의응답 로그, 최종 평가 점수, 피드백 등.

### 적용 방식 및 설정
현재 프로젝트는 `Prisma`라는 ORM을 사용 중이므로, 코드 수정을 최소화하고 연결만 변경하면 됩니다.
1.  AWS 콘솔에서 RDS(PostgreSQL) 생성 (초기: db.t3.micro 추천).
2.  `.env` 파일의 `DATABASE_URL`을 새로 생성한 AWS RDS 주소로 교체.
    (형식: `postgresql://[id]:[pw]@[rds-endpoint]:5432/[dbname]?schema=public`)
3.  `npx prisma db push`를 실행하여 RDS에 테이블 스키마 동기화.

*주의사항:* Vercel(서버리스)과 연동 시 DB 커넥션 초과(Connection Pooling) 방지 처리가 필수이며, AWS RDS Proxy나 Prisma Accelerate를 활용하거나 고정 IP 설정을 통해 보안과 안정성을 확보해야 합니다.

## 2. Vercel 배포 시나리오 및 서버리스 구조 한계
Vercel은 Next.js를 호스팅하는 최적의(가장 편한) 클라우드 플랫폼(CDN + Serverless)입니다.

### Vercel의 역할
*   사용자 화면(React) 컴포넌트는 글로벌 CDN 캐시로 전 세계에서 빠르게 렌더링.
*   백엔드(`app/api/...`) 코드는 "서버리스 함수(Serverless Functions)" 형태로 구동되어 사용자가 요청 시에만 실행되고 종료됨 (초기 인프라 유지비 제로).
*   GitHub Repository 연동만으로 PUSH 시 자동 배포(CI/CD) 및 고유 인터넷 주소(URL) 생성 기능을 지원합니다.

### 우리 프로젝트(AI 프로덕트) 기준 - Vercel의 치명적 한계점
1.  **Timeout(타임아웃) 제약:** Vercel의 프론트엔드/백엔드 통신은 Hobby 요금제 기준 API 응답 시간이 10초(Pro 요금제 적용 시 연장 가능)를 초과하면 무조건 `504 Timeout Error`로 렌더링 실패 처리됨. 우리는 긴 프롬프트를 사용할 가능성이 높으므로 치명적일 수 있습니다.
2.  **실시간 양방향 통신(WebSocket) 불가:** 추후 모의면접 기능 등에서 AI와 사용자가 음성/텍스트로 실시간 핑퐁을 하려면 웹소켓 지원이 필수이나, 서버리스 환경은 이를 유지할 수 없습니다.

## 3. 대안 클라우드 서비스 추천 (Long-running PaaS / IaaS)
AI를 활용한 복잡한 연산이 필요한 경우, 항시 켜져 있는(Long-running) 서버 환경이 더 안정적입니다.

*   **대안 1: Render 또는 Railway (PaaS)**
    *   **장점:** 타임아웃 제한이 없고, 웹소켓 등 실시간 통신 환경을 자유롭게 구축 가능. AWS RDS와의 연결 끊김(커넥션) 관리에도 이점이 있음.
    *   **단점:** Vercel 대비 캐싱 속도가 상대적으로 낮으며 프론트엔드 배포 편리성은 약간 낮음. 무/최소 비용 시 서버 수면 상태(Sleep)로 진입할 수 있음.
*   **대안 2: AWS App Runner 또는 EC2 + AWS RDS 직결**
    *   **장점:** RDS와의 동일한 네트워크 내부 통신으로 데이터 응답 속도가 최상임. 성능, 커스텀 한계가 없음.
    *   **단점:** 초기 인프라 설정(DevOps) 장벽이 높고, 트래픽에 따른 비용 관리가 까다로움.

### 👉 결론 및 향후 개발 전략 제안
1.  **MVP 검증 및 개발 초기 (현재): Vercel로 배포 유지**
    설정 제로, 손쉬운 CI/CD로 빠르게 화면을 띄우고 테스트하기 위해 초기 단계에서는 Vercel 배포를 권장합니다. AI 응답은 "Streaming(글자가 토큰 단위로 떨어지는 방식)" 기술로 타임아웃을 연장/우회하며 구현할 수 있습니다.
2.  **본격적인 AI 기능 고도화 / WebSocket 구현 시: Render 또는 AWS 이관**
    성능/시간 제약 문제가 지속적으로 발생하거나 모의면접 실시간 기능 단계에서는, 코드 수정 없이 호스팅 플랫폼만 Render, Railway, AWS 중 하나로 옮겨서 안정적으로 운영하는 것을 권장합니다.
